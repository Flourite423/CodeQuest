\chapter{需求分析}

\section{总体需求分析}

本系统面向前端初学者构建移动客户端学习环境，目标是在移动设备上形成课程学习、代码练习、闯关挑战、每日任务、社交激励与个人成长相结合的完整学习闭环。根据前期调研与系统目标，系统由移动客户端、服务端与管理后台端组成，其中移动客户端承担主要学习活动，服务端负责统一业务处理与数据支撑，管理后台端负责内容维护、运营支撑与数据管理。该类系统既需要满足移动学习场景下的便捷访问需求，也需要兼顾编程教育中实践反馈及时、任务组织明确和学习持续性较强等特点。国内学者方芳\cite{fang2007} 较早从理论基础角度对教育游戏进行了系统分析，为游戏化学习机制的本土化应用提供了参考框架\cite{robins2003,heines2001,sommerville2015}。

从学习目标来看，系统不仅要支持前端基础知识的呈现和阅读，还应支持学习者通过编码练习和挑战任务完成知识内化，并借助成长反馈与社交激励增强持续学习意愿。系统需求因此不能只停留在内容展示层面，而需要覆盖学习行为记录、任务组织、结果反馈和成长激励等多个维度\cite{hamari2014,ryan2000,kalmpourtzis2018}。前端基础知识包括 HTML 标记语言、CSS 样式语言和基础页面布局等内容，这些知识既具有明确的语法规则，又需要通过大量实践才能熟练掌握，系统需求因此特别强调理论与实践的结合。

目标用户主要是前端初学者，他们可能具备一定的计算机使用经验，但对 Web 开发技术尚不熟悉。这类用户在学习过程中容易因概念抽象、错误难以定位或缺乏及时反馈而产生挫败感，系统因此在交互设计上需要注重引导性、反馈的及时性和错误提示的可理解性。初学者通常没有固定的学习时间和场所，移动学习场景正好契合其利用碎片时间学习的需求，但这对系统的离线能力和状态恢复能力也提出了更高要求。

系统边界方面，本文聚焦于学习应用本身的设计与实现，不包括完整的在线代码执行环境、大规模用户行为分析系统或商业化运营平台。AI 辅助功能限定在错误解释和提示层面，不扩展为完整的智能辅导系统。社交功能限定在好友、动态和排行榜等基础互动形式，不扩展为实时通信或协作编程等复杂场景。这些边界界定使系统需求保持在可实现的范围内，同时为核心学习闭环的完整性提供保障。

\section{技术需求分析}

运行平台方面，移动客户端需要支持 iOS 与 Android 智能手机，适配不同尺寸屏幕并保持基础交互一致性；管理后台端主要面向桌面浏览器环境，用于内容维护与配置操作；服务端需要具备面向多端的统一 API 提供能力。由于系统存在移动客户端学习场景，网络环境不能假设始终稳定，应用需要具备一定的离线感知与本地缓存能力\cite{liang2018,jiao2023}。

移动客户端的技术需求还包括对移动设备特性的适配：需要处理不同屏幕尺寸和分辨率的布局适配问题，需要在虚拟键盘弹出时调整界面元素位置，需要处理应用前后台切换时的状态保存和恢复。在性能方面，移动客户端需要控制内存占用和电量消耗，避免长时间使用导致设备过热或电量快速下降。在存储方面，合理利用本地存储空间并对缓存数据进行有效管理，能够避免因存储膨胀影响应用性能。

系统协同方面，服务端接口必须能够同时支撑移动客户端与管理后台端的访问需求，并通过统一响应结构保障调用一致性。接口文档需要作为开发与联调的共同依据，以减少多端协作中的理解偏差。对移动客户端而言，还需要具备认证信息持久化、进度缓存、弱网回退与错误处理能力，以保证学习流程的连续性\cite{brusilovsky2007,papamitsiou2014,sommerville2015}。在人工智能与教育结合的前沿探索方面，Ji 等人\cite{ji2022} 对知识图谱的综述显示了结构化知识表示在学习系统中的潜在价值，Holstein 等人\cite{holstein2018} 对混合现实教学工具的研究为 AI 增强型课堂提供了实证参考，Ayad 和 Alsayoud\cite{ayad2024} 对 ChatGPT 提示工程的研究则提示了大型语言模型在学习支持系统中的新可能。

服务端的技术需求还包括对并发访问的处理能力、对数据库操作的事务保障、对异常情况的容错处理等。虽然本文不涉及大规模性能优化，但服务端需要保证在正常负载下能够稳定响应请求，在异常情况下能够给出合理的错误信息而非直接崩溃。管理后台端的技术需求则侧重于内容编辑的便利性、数据展示的清晰性和操作反馈的及时性，使运营人员能够高效地完成日常维护工作。

\section{功能需求分析}

\subsection{移动客户端功能需求}

结合系统目标与功能规划，移动客户端功能可归纳为以下七个方面。

\begin{enumerate}
  \item 账户管理功能，包括注册、登录与个人资料维护，用于建立用户身份与学习档案。
  \item 课程学习功能，包括课程列表浏览、课程详情查看、章节阅读和学习进度跟踪，用于支撑基础知识学习过程。
  \item 编码练习功能，包括练习详情查看、代码提交与结果反馈，用于连接知识理解与实践操作\cite{robins2003,heines2001,guo2013}。
  \item 挑战相关功能，包括闯关挑战、星级评价、奖励发放与每日挑战，用于将学习活动组织为具有目标感与节奏感的任务链。
  \item AI 辅助功能，用于在学习者练习受阻时提供错误解释与必要提示，其范围限定在辅助解释层面，不扩展为完整的代码生成或个性化推荐系统。
  \item 社交互动功能，包括好友、动态与排行榜，用于提供外部激励与学习氛围。
  \item 成长反馈功能，包括经验值、徽章、等级与统计信息，用于沉淀学习成果并强化用户成长感知\cite{koedinger1997,woolf2010,liang2018,jiao2023}。
\end{enumerate}

\subsection{服务端功能需求}

服务端作为统一的业务处理与数据管理中心，为移动客户端和管理后台端提供接口支撑，其功能需求可归纳为以下七个方面。

\begin{enumerate}
  \item 认证鉴权功能，包括学习者注册与登录、管理员登录、令牌签发与刷新、Token 校验中间件，用于建立系统级的身份验证与权限控制体系。
  \item 课程与内容服务，包括课程列表与详情查询、章节内容获取、练习题目查询，同时为管理后台端提供课程、章节与练习的增删改查接口。
  \item 练习提交与评测，包括提交记录的创建与查询、评测结果的生成与反馈，并根据提交结果更新学习者的经验值与进度统计。
  \item 挑战与奖励管理，包括闯关挑战列表查询与参与、星级评定与奖励结算、每日挑战题目生成与完成记录、奖励摘要与经验值账本查询。
  \item AI 辅助服务，包括帮助请求的创建与历史查询，通过接入大语言模型 API 生成分层提示内容并返回移动客户端。
  \item 社交与排行功能，包括好友关系的建立与维护、好友动态的聚合与推送、多维度排行榜的排名计算与展示\cite{sommerville2015,campos2022}。
  \item 进度管理功能，包括课程与章节级别的学习进度记录、进度查询与更新、经验值发放与徽章检查触发。
\end{enumerate}

\subsection{管理后台端功能需求}

管理后台端面向运营人员，承担内容维护、用户管理与运营支撑职能，其功能需求可归纳为以下六个方面。

\begin{enumerate}
  \item 数据看板功能，以统计卡片形式展示平台核心运营指标，包括总用户数、总课程数、活跃用户数与待处理审核数。
  \item 课程与题目管理，支持课程的增删改查与发布状态控制，支持章节的排序调整与练习题目的类型、难度和状态管理。
  \item 挑战管理，支持闯关挑战的增删改查，包括挑战标题、难度等级、经验值奖励与发布状态的配置。
  \item 用户管理，支持平台用户的查看与状态管理，包括按昵称或邮箱搜索、按状态筛选，以及账户的封禁与解封操作。
  \item 内容审核，支持对学习者提交内容的审核流程，包括待处理与已处理审核的分标签展示，支持批准与拒绝操作并记录审批原因。
  \item 公告与配置管理，支持平台公告的增删改查与发布控制，以及站点基本配置项的管理。
\end{enumerate}

\section{非功能需求分析}

系统应保证学习者能够以较短路径完成主要学习任务，界面反馈清晰、状态切换明确，减少因复杂操作造成的学习中断。遵循最小操作路径原则，将常用功能置于主界面直接可达位置，避免深层嵌套的导航结构，使学习者能够在较少步骤内进入目标学习内容。错误提示应使用学习者能够理解的语言，避免展示技术性错误代码或堆栈信息\cite{heines2001,guo2013}。

响应性方面，课程加载、页面切换、请求返回与错误反馈需要具备基本可接受的交互速度，并能在网络异常时给出可理解的提示。课程列表和章节内容的加载时间应控制在用户可接受的范围内，代码提交后的结果反馈应尽可能及时，避免学习者因等待时间过长而产生焦虑。在网络异常时，系统应明确告知用户当前网络状态，并提供重试或稍后操作的选项。

可维护性依赖于各端职责清晰、接口结构统一。代码组织遵循模块化原则，各功能模块之间的耦合度应控制在合理范围内，使单个模块的修改不会对其他模块产生意外影响。文档覆盖关键设计决策和接口定义，使新加入的开发者能够快速理解系统结构。

一致性方面，系统通过统一契约文档控制接口字段、响应包络与权限边界，保证多端协作不因接口漂移而失效。移动客户端的界面风格应保持一致，相同功能的交互方式应在不同页面中遵循相同模式，减少学习者的认知负担。服务端对同类请求的处理逻辑应保持一致，避免因特殊处理导致意外行为。

可靠性依赖于本地存储和离线回退策略：关键操作如代码提交应在本地保存副本，网络恢复后自动重试提交，避免因网络抖动导致学习成果丢失。服务端则通过对关键操作采用事务保障，避免部分更新导致的数据不一致。

扩展性方面，系统通过管理后台端提供了课程、挑战与题目的管理接口，支持在不修改代码的前提下新增课程内容、题目配置与挑战规则，为后续内容的持续扩充保留了管理层面的操作空间。服务端数据模型在设计时考虑了字段的可扩展性，新增属性不会影响现有功能的正常运行。本文只围绕当前实现范围展开，不对后续版本的具体功能进行详细设计\cite{sommerville2015}。

兼容性方面，移动客户端需要适配主流 iOS 和 Android 系统版本，对较旧版本系统提供降级体验。服务端接口设计遵循向后兼容原则，避免因接口变更导致旧版本移动客户端无法正常使用。管理后台端需要兼容主流桌面浏览器，确保运营人员能够在常见浏览器环境中完成内容管理工作。

可访问性方面，系统遵循基本的可访问性设计原则，包括界面元素具有足够的对比度、交互控件具有适当的尺寸、状态变化具有明确的视觉反馈等。虽然本文不将可访问性作为核心研究目标，但这些基础考虑有助于扩大系统的适用人群。

国际化与本地化方面，当前系统主要面向中文用户群体，界面语言和内容均以中文呈现。结合本科毕业设计的实现边界，本文当前并未对多语言支持进行具体实现，而是将其作为后续可能的扩展方向保留讨论空间。

\section{安全性需求}

安全性方面，系统需要在多个层面建立基本的防护能力。身份认证层面，用户密码在服务端不得以明文形式存储，必须经过单向哈希（如 bcrypt 或 argon2）处理后再持久化，防止数据库泄露时密码直接暴露。传输安全层面，移动客户端与服务端之间、管理后台端与服务端之间的通信需要支持 HTTPS 加密，降低敏感数据在传输中途被截获或篡改的风险。接口访问控制层面，所有涉及学习者个人数据、课程内容、提交记录和奖励信息的接口必须经过 Token 校验，未授权请求应返回统一的 401/403 响应而非暴露具体资源信息。

输入安全层面，服务端需要对所有外部输入进行基础校验，包括路径参数的格式校验（如 UUID 格式）、查询参数的边界校验（如分页参数不超过最大值）、请求体的必填字段检查和类型校验。对于数据库操作，系统必须通过参数化查询（如 SQLx 的预编译语句）构造 SQL，杜绝字符串拼接方式，以防止 SQL 注入攻击。管理后台端创建课程、题目和挑战等内容的操作应受到角色校验，确保只有具备管理权限的账户才能执行写入操作。

数据隔离层面，学习者侧接口不得返回管理侧敏感字段（如正确答案标记、隐藏测试用例、用户审计字段等），管理后台端接口也不得将学习者私有数据暴露给非授权的后台操作员。这些措施虽然不构成独立的安全认证体系，但它们为系统的数据安全提供了必要的工程底线，确保系统在用户信息保护方面不低于同类学习应用的基本水准。
